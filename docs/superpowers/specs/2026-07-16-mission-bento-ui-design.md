# Mission Bento UI — Design Specification

**Date:** 2026-07-16 · **Risk class:** Normal (UI-only; no auth/crypto/migrations/forecast-engine changes)
**Approved design reference:** `docs/mockups/06-mission-bento.html` in the main checkout (untracked design artifact; the token tables below are the durable record).
**Decision trail:** user selected "Mission Control" direction with bento overview (2026-07-16), then: uniform tiles, 3 per row, smaller verdict headline, no action queue on the overview, plus six UX upgrades (calm state, stale-baseline warnings, scenario ghost line, host lifecycle Gantt, interactive rail, urgency sort).

## 1. Problem and outcome

The current UI is a generic 2020 admin template with two near-duplicate pages (Overview, Clusters). The redesign turns the app into a single **fleet console**: one screen that answers "what do I have to order, and when" at a glance, with cluster detail as a slide-in panel instead of a separate page. Forecast/procurement data drives every element; no new backend endpoints are required (order-by dates, scenario overlays, host lifecycle dates and thresholds all exist in the API today).

## 2. Information architecture

| Route           | Before                                   | After                                                                                                                  |
| --------------- | ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `/`             | Overview (KPI tiles, tile grid, heatmap) | **Fleet console**: order-by rail, fleet verdict, cluster tile grid (3-up), add-cluster + show-archived controls        |
| `/clusters`     | Clusters list (table + grid + KPIs)      | Redirect to `/`                                                                                                        |
| `/clusters/new` | Dead placeholder                         | Deleted                                                                                                                |
| `/clusters/$id` | Detail page                              | Same URL renders the fleet console **with the detail slide-in panel open** (deep-linkable; Esc/close navigates to `/`) |
| `/settings`     | Settings page                            | Unchanged structurally; restyled by tokens                                                                             |
| `/login`        | Login                                    | Unchanged structurally; restyled by tokens                                                                             |

Chrome: the sidebar, breadcrumbs, and mobile nav drawer are removed. A single topbar carries: brand (links `/`), search/⌘K trigger, Settings link, theme toggle, user menu. Command palette: "Go to overview"/"Go to clusters" collapse into "Go to fleet" (`g o`; `g c` removed), per-cluster entries keep navigating to `/clusters/$id` (which now opens the panel). `g s` unchanged.

_Amended 2026-07-19 (#243 Part B):_ the `/settings` row's "unchanged structurally" was a before/after note against the pre-redesign page, not a prohibition on later restructuring, and the page had grown into one flat scroll with no in-page navigation (Medium finding, `docs/uiux-audit-2026-07.md`). The five panels are now grouped under three labelled, anchor-linked `<section>`s — **Forecasting** (Forecast thresholds, Categories), **Inventory** (vCenter connections, Add cluster), **Access** (Authentication, gated the same as before) — with a plain in-page nav above them (`#section-forecasting` / `#section-inventory` / `#section-access`; these are static jump links, not the `requestAnchorFocus` deep-link contract `add-cluster` uses, since there is nothing to re-invoke). Each panel's own heading demotes from `h2` to `h3` so the outline under the new section `h2`s never doubles a level. The `Settings` h1 adopts `font-display`/`text-display` (§3 tokens, previously unused anywhere) in place of the arbitrary `text-[26px]` Inter fallback, and the `'CONFIGURATION'` eyebrow above it is dropped. Settings form cards (Forecast thresholds, Categories, the cluster panel's Settings tab) are width-capped at `max-w-2xl` with numeric inputs at an intrinsic `w-24`, matching the `w-full max-w-sm` capping idiom `ScenarioPaneBody` (§5) already uses for the same reason — a percentage or a weeks count read as broken stretched across an uncapped card.

_Amended 2026-07-19 (#243):_ the `--glass-*` rows above are new — the tokens behind the **one sanctioned glass surface**, the scenario controls card (`.scenario-card`; see the §5 amendment and CLAUDE.md's house-style exception). Fill/fallback derive from the slate surfaces (`--card` with alpha in dark; white with alpha in light); the border is a light hairline in dark and a `--border-strong` mix in light, and is mandatory — it is the card's only edge in forced-colors mode. The fallback is near-opaque and MUST pass AA on its own: it serves engines without `backdrop-filter` AND `prefers-reduced-transparency` users (that media query is not Baseline — no Firefox). One recorded deviation from the issue's reference recipe: the card's lift shadow is the themed `--overlay-shadow` (plus the inset lit top edge), not the fixed `0 8px 32px rgb(0 0 0/.35)`, which out-shadowed every other light-theme overlay. A second recorded constraint from the worst-case contrast check: in **light theme below `lg`** the card sits on the sheet's black/40 scrim, where the 70% white fill composites to ~#e0e0e0 and the 10px `--fg-subtle` labels measure ~3.7:1 — so the sheet body uses `--glass-fallback` there (per the issue's own rule: raise tint alpha, never blur). Dark keeps full glass under its scrim (≈5:1+) and light keeps it at `lg`+ (no scrim, ≈4.8:1). The 1024px media query in `styles.css` must stay in lockstep with `PANE_SIDE_BY_SIDE_QUERY`. _Amended 2026-07-19 (#243 review):_ that lockstep is now guaranteed by construction — all Tailwind breakpoints are pinned to their px defaults in the `@theme` block (`--breakpoint-sm: 640px` … `--breakpoint-lg: 1024px` …). Tailwind v4's rem-based defaults resolve media queries against the user's **browser default font size**, so `lg:` utilities desynced from the px `matchMedia` constants (`PANE_SIDE_BY_SIDE_QUERY`, `isWide`): at a 20px default font, a 1024-1280px band rendered the sub-`lg` sheet layout (scrim + near-viewport-wide card) over a live, non-`inert` content column — the focus-obscured condition the containment exists to prevent — and this AA fallback failed to fire. Any new JS media query MUST use the same px values as the utilities it pairs with.

Radii: `--radius: 8px`, `--radius-card: 14px`, `--radius-modal: 16px`. Focus: global two-layer ring — `outline: 2px solid var(--steel); outline-offset: 2px` plus a `box-shadow` separator in the surface color. Motion: hover transitions 120 ms; panel enters 280 ms `cubic-bezier(0,0,.38,.9)`, exits 200 ms `cubic-bezier(.4,0,1,1)`; existing `prefers-reduced-motion` kill-switch stays. Status is never color-alone (chip text/icons accompany every state). _Amended 2026-07-19 (#243):_ the detail panel's enter/exit motion is retired — the panel opens and closes **instantly** (open/close render on the next frame; no close delay). Frequent, user-triggered transitions get the shortest/subtlest motion or none (NN/g, Apple HIG), and the fullscreen takeover genuinely reads as a page. The 280/200 ms curves above remain in force for the **Scenario pane only** — a deliberate, recorded motion asymmetry (panel = high-frequency navigation; pane = occasional mode change) — and the scenario card animates transform/opacity only, never a blur radius.

## 4. Fleet console composition (top → bottom)

1. **Topbar** — brand, ⌘K search trigger, Settings link, theme toggle, user menu.
2. **Order-by rail** — full-width 12-month strip. NOW line at left (steel, labeled). One tick per cluster with a non-null `forecast.procurement.orderByDate` inside the window, labeled `<cluster> · <MMM D> · <urgency word>`; the earliest tick gets the shaded 90-day/lead-time zone. Tick urgency tone: crit if order-by ≤ 28 days or past (matches `deriveProcurementKpi` URGENT_DAYS), warn if ≤ 90 days, muted otherwise. Empty state (no order-bys): centered ✓ + "No order-by dates in the next 12 months". Hover/focus of a tick highlights its tile (and vice versa) via `data-cluster-id` linking. _Amended 2026-07-18 (#218):_ the lead-time zone is respecified — its _presence_ is a property of the **rail** rather than of the earliest tick, and it is **labeled**. (Its width is still read off the earliest tick's `leadTimeWeeks`, which is simply where the tenant-wide setting is carried onto each item; the value is uniform across a fleet.) (1) **Width** stays the configured tenant lead time (`procurement.leadTimeWeeks × 7`, default 8 wk = 56 d), measured forward from NOW and clamped to the 365-day window; the mockup's "90-day" zone was that demo dataset's lead-time value, not a constant, so "the shaded 90-day/lead-time zone" above reads as the lead-time zone throughout. (2) **It renders whenever the rail has ticks**, dropping the old `earliest.days >= 0` guard, which hid the zone exactly when it mattered most: with any overdue order-by the whole highlight vanished, though "anything ordered today lands after this window" is most urgent precisely then. It is suppressed only when the rail is empty or the tenant configures a 0-week lead time (`procurementLeadTimeWeeks` allows 0), where a zero-width zone and a "LEAD 0 D" label would both be noise. (3) **A `LEAD {n} D` micro-label** sits at the zone's right edge (mono 9 px, `translateX(-100%)` so it reads inside the zone against the dashed boundary), because the 9 %→3 % accent wash alone was too faint to notice and left the width no legible meaning. The label is decorative (`aria-hidden`) and the **rail-header hint carries the meaning in text** — it now interpolates the same number ("shaded = inside 56-day lead time · tick = last safe order date"), so the zone is never signalled by shading alone. **Token:** the label uses a new `--accent-label` — full-strength `--accent` in light, 75 % `--accent` in dark. Contrast was measured in-browser against the wash-over-card the text actually sits on: the mockup's flat 75 % softening clears AA in dark (5.6:1) but reaches only 2.97:1 in light, far below the 4.5:1 floor for 9 px text, so light keeps the accent undimmed (4.67:1) and only dark takes the softening. This mirrors `--accent-soft`, which already varies per theme (10 % light / 15 % dark). _Amended 2026-07-19 (#243 Part B):_ two more changes. (1) The heading is renamed from the spec-internal "Order-by rail" to the user-facing "Order deadlines — next 12 months" ('rail' is a design-system term, not a user concept). (2) The empty state collapses to a compact single-row strip — heading and the ✓ sentence rendered inline, the fixed 86px tick area and the month axis both hidden — instead of restating the same all-clear the verdict headline already states, ~170px below it. The header-row hint's empty-state copy changes from describing the (invisible, not-yet-relevant) lead-time zone to describing what a tick will mean once one exists ('each mark = a cluster's last safe order date'); the #218-mandated populated hint text is unchanged.
3. **Fleet verdict panel** — display-font headline computed from fleet state: urgent form "Fleet runway is **N mo** — **<cluster>** needs an order by **<MMM D>**." (from `fleetRunwayToWarn` + `earliestOrderByFromFleet`); all-clear form "Fleet is healthy — no orders due before **<horizon>**." Sub-deck instrument row: utilization (linear bullet meter with warn/crit ticks — no radial gauges), headroom GB, clusters · hosts count, open-order count (order-bys within 90 days), and **Baselines** (count of clusters with `baselineDate` older than 90 days; warn-toned when > 0, "all fresh" otherwise). _Amended 2026-07-19 (#243 Part B):_ the meter's warn/crit threshold ticks each carry a 1px halo in the card surface color (`shadow-[0_0_0_1px_var(--card)]`) and are fully opaque — in dark theme `--warning` and `--accent` resolve to the same hex, so the previous 70%-alpha warn tick vanished into the amber fill exactly at warn breach. Crit is additionally taller than warn (protrudes 4px vs 2px past the track), so the severities differ by shape, not hue alone (WCAG 1.4.1). Applies to every `BulletMeter` instance (fleet verdict row + panel KPI strip). _Amended 2026-07-19 (#243 Part B, second round):_ three more changes. (1) The headline's non-link `<strong>`s (the numeral, date, "healthy"/"unknown") drop the underline they shared with the cluster-name `<Link>` — plain text read as an identically-styled dead link, since the urgent branch's real link uses the exact same treatment. Color and weight now carry the emphasis; the underline is reserved for the one real link. (2) The all-clear form is rescoped from "no orders due before **<horizon>**" to "no orders due in the **N-month** forecast window" (N = the real aggregated series length, `summary.fleetMonths.length`) — the old wording phrased a horizon-edge date as a guarantee it couldn't back up (a breach shortly past the horizon, minus lead time, could still demand an order before that date). When no forecast series has loaded at all (`fleetMonths` empty — reachable when clusters carry recorded capacity but every per-cluster forecast lookup misses), the phrase degrades to the unnumbered 'the forecast window' instead of printing '0-month'. (3) Instrument-row dividers are now drawn structurally instead of as standalone flex items: a `border-l` on **every** instrument (sm+ only), with the row offset `-mx-6` inside an `overflow-hidden` wrapper so whichever instrument begins a visual row has its leading rule clipped. Bordering only `*+*` (every instrument but the first) is **not** sufficient — CSS cannot detect a row start, so that form merely relocates the artifact from the end of row one to the start of row two; the offset-and-clip is what makes wrapping safe at both ends of every row.
4. **Cluster tile grid** — 12-col grid, tiles `span 4` (3-up ≥1280 px, 2-up ≥820 px, 1-up below). One uniform tile per non-archived cluster, **sorted by order-by date ascending, nulls last, runway tiebreak**, with a "sorted by order-by date" micro-note. Tile contents: name + status chip (`utilStatus`) + flag chips (archived, license/lifecycle notes are out of scope — flags shown: event within window, stale baseline); runway numeral line ("9 MO to warn 2027-04" / "24+ MO no breach"); order-by chip ("ORDER BY 2026-12-28 · IN 5 MO", tone by urgency; "— · no order needed" when null); one-line verdict sentence; baseline chip ("BASELINE 2026-06-20", warn variant "⚠ BASELINE 128 D OLD" past 90 days); a compact Recharts forecast chart — **shared 0–125 %-of-capacity y-scale across all tiles**, warn/crit hairlines + 100% capacity line, consumption solid to the current month then dashed (forecast), labeled breach dot and order-by marker, hover tooltip. _Recorded exception (2026-07-17):_ compact tile charts omit standalone event markers, matching the approved mockup; events surface in breach sub-labels on tiles and as full markers on the detail chart. Reason: legibility at tile scale; risk: low (detail chart carries them); approver: design lead per approved mockup; review: at final branch review. Clicking/Enter navigates to `/clusters/$id`. _Amended 2026-07-17:_ the runway numeral example above reads "9 MO to warn 2027-04" (was "to crit") — the numeral always tracks the warn-threshold breach month (`computeRunway`'s `breachLabel: 'warn'` case), matching `runwayToWarn`; crit only appears in the past-breach sub-line/verdict text once the fleet is already past warn. _Amended 2026-07-18 (#224):_ two changes to the compact tile chart. (1) **Axes are now labeled** — the x-axis shows short month names (`formatMonthShort`, `interval="preserveStartEnd"` so they thin at tile width) and the y-axis shows a few percent ticks; margins/axis sizing were adjusted from the previous hidden-axes layout. (2) **The shared y-scale is tightened from 0–125 % to 40–125 %** — real clusters sit well above 40 %, so the old scale wasted its bottom third and squeezed the warn/crit band into a top sliver. The window stays **fixed and identical across all tiles** (the comparability invariant is preserved), enforced with Recharts `allowDataOverflow` so out-of-range data can never stretch the axis: below-floor values (notably zero/unknown-capacity synced clusters at 0 %, per #198) are clamped to the 40 % floor so their line stays visible pinned to the bottom edge, while the tooltip still reports the true percentage; above-ceiling values (>125 %) clamp to the top as before. The warn/crit **hairlines are clamped into the window too**: Recharts' `ReferenceLine` defaults to `ifOverflow="discard"`, so a threshold configured below the floor (`percentSchema` allows 0.01) would silently render no line at all while the breach dot still pinned to the floor and the aria-label still announced the threshold — pinning the hairline to the 40 % edge instead keeps the tile self-consistent and honest (the entire visible window is then at or beyond that threshold), with the aria-label still reporting the true percentage. A clamped hairline is **marked off-scale** by switching its dash from `4 3` to a finer `1 2`, so a line pinned at the 40 % edge cannot be misread as a threshold genuinely configured at 40 % (pattern, not color, so the cue survives color-vision deficiency). When **both** thresholds clamp to the same edge they would otherwise render exactly coincident — hiding warn permanently under crit while still implying two distinguishable bands — so a single merged off-scale hairline is drawn in the crit color instead, with the aria-label continuing to report both true percentages. **Color deviation:** the consumption line moves off amber to a dedicated violet (`--chart-consumption` = `#7c3aed` light / `#c084fc` dark, split from `--accent`). House style reserves amber for brand+data, but amber is double-booked as the warn-threshold color (`--warning`), so in dark theme the usage line and the warn hairline were the identical hex — indistinguishable. The warn threshold keeps amber (its semantic status color); the consumption line takes violet, which clears warn amber, crit red, success green, and the steel interaction hue in both themes. `--steel` was explicitly rejected (reserved for interaction/focus, and already used for the tile's order-by marker). This also recolors the cluster-detail `ForecastChart`, which shares `--chart-consumption`. _Amended 2026-07-19 (#243 Part B):_ two chips become conditional instead of always-on. The order-by chip ("— · no order needed" above) now renders only when there is a real order-by date or the capacity-unknown case ("— · ORDER STATUS UNKNOWN") — a healthy, known-capacity tile omits it entirely, since the OK badge, the runway numeral, and the verdict sentence already state the same all-clear and the placeholder chip was a fourth restatement. The baseline chip likewise renders only in its stale/warn variant ("⚠ BASELINE N D OLD"); a fresh baseline's plain chip is omitted, since the verdict panel's "BASELINES ✓ all fresh" instrument already covers that case fleet-wide.
5. **Console controls** — "+ Add cluster" (AdminOnly, existing `CreateClusterDialog`) and "Show archived" toggle (archived clusters append as muted tiles, runway shown as "—"). _Amended 2026-07-17:_ the controls render as a slim toolbar row between the rail and the verdict panel (not below the grid); the numbered list orders content, not strict visual position for this item. The `ClusterTable`, `FleetUtilizationHeatmap`, `FleetClusterGrid`, `KpiTile` row and `ClusterListCard` are retired. _Amended 2026-07-18 (#223):_ "+ Add cluster" is removed from the console entirely — manually adding a cluster is a configuration task and now lives in an AdminOnly Settings panel (below vCenter connections), leaving "Show archived" as the only console control. The empty console shows a large AdminOnly Add-cluster call-to-action in the tile-grid area that links to Settings (viewers get a plain explanation instead); the ⌘K "Create cluster" action navigates to `/settings` and is hidden from viewers (it previously dispatched `lcm:open-create-cluster`, now removed). _Amended 2026-07-18 (#233, review follow-up to #223):_ both Settings hand-offs above are **deep links**, not bare `/settings` navigations. The Add-cluster panel is the fourth panel down the Settings page, so landing at the top of `/settings` left both the viewport and keyboard focus nowhere near the control that had just been promised. (1) The ⌘K action is relabelled **"Add cluster — Settings"** — it names its destination rather than the retired in-place dialog — and navigates to `/settings` with hash `add-cluster`; cmdk `keywords` (`create cluster`, `new cluster`, `add cluster`) keep the pre-#223 muscle memory searchable. (2) The Settings panel carries `id="add-cluster"` (`ADD_CLUSTER_HASH`) and, when that hash targets it, scrolls itself into view (`behavior: 'smooth'`, `'auto'` under `prefers-reduced-motion`) and focuses its "+ Add cluster" trigger with `preventScroll`, so the focus call does not jump-cut past the smooth scroll. (3) The empty-console CTA links to the same `/settings#add-cluster`. The action stays AdminOnly/hidden from viewers, and the deep-link effect lives inside the admin-gated card so it only runs where the panel actually renders. **Two triggers are load-bearing, not belt-and-braces:** re-invoking the action while the URL is _already_ `/settings#add-cluster` produces an identical location, so a hash-keyed effect never re-runs and the action would be a silent no-op — an explicit focus request (`requestAnchorFocus` / `useAnchorFocusRequest`, `src/lib/anchors.ts`) carries that repeat case, while the hash carries first mount, reload, and shared links. _Amended 2026-07-19 (#243):_ the stranded "Show archived" toggle row is retired. Mitigations for the popover indirection's known discoverability cost: the trigger carries an active-count badge (`Filter · 1`) and a steel active tone while any non-default filter is on, so a mixed view is always explainable from the toolbar. ## 5. Detail slide-in panel (`/clusters/$id`)

Fixed right panel, 58 vw (min 600 px; 100 vw < 1100 px), entering/exiting per the motion spec; the console remains beneath. Esc or ✕ navigates to `/`; focus is moved into the panel on open and restored on close. Content preserves the current detail page's functionality, restyled: _Amended 2026-07-17:_ the panel opens as a fullscreen takeover — 100 vw at every width (the 58 vw / min-600 px / 1100 px-breakpoint geometry and the partial-panel left border/shadow are retired) — user decision after using the shipped UI; the slide-in/out motion, Esc/✕, focus management, and inert console beneath are unchanged. _Amended 2026-07-18 (#225/#226):_ this supersedes the "Esc or ✕ navigates to `/`" sentence above and the 2026-07-17 "Esc/✕, focus management … unchanged" clause. (1) **✕ → Back.** The header's ✕ Close control is now a labeled Back button (`BackButton`: `ArrowLeft` icon + "Back" + a decorative `Esc` `<kbd>` hint, both icon and `<kbd>` `aria-hidden` so the accessible name is exactly "Back"); dismissal behavior and focus semantics are otherwise unchanged — it still runs the exit animation, announces "Cluster … detail closed." on the polite live region, then navigates to `/`. ## 6. Accessibility & quality floor

WCAG 2.2 AA: global `:focus-visible` two-layer ring (≥2 px, ≥3:1 on every surface); interactive targets ≥24 px (invisible hit-area padding on rail ticks); tile grid keyboard-operable (tiles are links/buttons); panel focus management + `aria-live` announcements for open/close and scenario deltas; status chips always pair color with text; charts get halos on labels crossing marks; reduced-motion honored everywhere. Both themes ship together and every screen must pass in both.
